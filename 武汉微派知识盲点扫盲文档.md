## 盲点 1：Unity 内存分区体系 🔴 高优先级

### 核心问题

你不知道贴图等 Unity 内置对象存在哪块内存区域，误以为在托管堆。

### 扫盲：Unity 三层内存架构

```
┌─────────────────────────────────────────────────────┐
│                    托管堆 (Managed Heap)              │
│  C# new出来的对象、字符串、数组、List、Dictionary     │
│  GC自动回收，但会产生碎片和停顿                        │
│  大小不固定，会动态扩容                               │
├─────────────────────────────────────────────────────┤
│                    原生堆 (Native Heap)               │
│  贴图、Mesh、Shader、AudioClip、Prefab实例            │
│  手动管理（Resources.UnloadAsset / AssetBundle.Unload）│
│  不受GC控制                                           │
├─────────────────────────────────────────────────────┤
│                    栈 (Stack)                         │
│  值类型局部变量、方法参数、返回地址                    │
│  方法结束自动释放，大小固定（通常1MB）                │
└─────────────────────────────────────────────────────┘
```

### 常见误区

| 误区 | 真相 |
|------|------|
| "所有东西都在托管堆" | 只有 C# 管理的对象在托管堆，原生资源在原生堆 |
| "Resources.Load 后就在托管堆" | 返回的引用在托管堆，但实际对象（贴图等）在原生堆 |
| "GC 能释放贴图" | GC 只能释放托管堆，原生资源必须手动释放 |
| "Destroy 后内存立即释放" | GameObject 立即销毁，但引用它的资源（如贴图）要看引用计数 |

### 面试应答模板

> "Unity 内存分三层。托管堆由 GC 管理，存 C# 对象。原生堆存 Unity 底层资源（贴图、Mesh、Shader），需要手动调用 `Resources.UnloadAsset` 或 `AssetBundle.Unload` 释放。栈存值类型局部变量，自动管理。贴图这类资源在原生堆，不受 GC 控制——这也是为什么频繁加载/卸载贴图时，如果不主动调用卸载接口，就会内存持续增长。"

## 盲点 2：通用泛型内存池设计 🔴 高优先级

### 核心问题

面试官追问"通用内存池需要继承什么"时，你完全答不上来。承认只用过类型确定的内存池。

### 扫盲：泛型内存池的两种主流设计

### 方案一：泛型类 + Action 回调（最简单，推荐）

```csharp
public class GenericPool<T> where T : class, new()
{
    private readonly Stack<T> _available = new Stack<T>();
    private readonly Action<T> _onReturn;  // 归还时的清洗逻辑

    public GenericPool(Action<T> onReturn = null, int prewarm = 0)
    {
        _onReturn = onReturn;
        for (int i = 0; i < prewarm; i++)
            _available.Push(CreateNew());
    }

    private T CreateNew() => new T();

    public T Get()
    {
        return _available.Count > 0 ? _available.Pop() : CreateNew();
    }

    public void Return(T item)
    {
        _onReturn?.Invoke(item);  // ← 数据清洗在这里
        _available.Push(item);
    }

    public int AvailableCount => _available.Count;
}

// 使用示例
var bulletPool = new GenericPool<Bullet>(b => {
    b.Velocity = Vector3.zero;
    b.IsActive = false;
    b.Damage = 0;
}, prewarm: 20);

Bullet bullet = bulletPool.Get();
// ... 使用 bullet ...
bulletPool.Return(bullet);
```

### 方案二：接口约束（最规范，Unity 常用）

```csharp
// 池化对象必须实现的接口
public interface IPoolable
{
    /// 从池中取出时调用
    void OnSpawned();

    /// 归还池时调用
    void OnDespawned();
}

// 所有可池化的类实现这个接口
public class EffectParticle : MonoBehaviour, IPoolable
{
    public ParticleSystem ps;
    public float duration;

    public void OnSpawned()
    {
        gameObject.SetActive(true);
        ps.Play();
    }

    public void OnDespawned()
    {
        // 清洗数据
        ps.Stop();
        ps.Clear();
        duration = 0;
        gameObject.SetActive(false);
    }
}

// 通用对象池
public class UnityPool<T> where T : Component, IPoolable, new()
{
    private readonly Stack<T> _pool = new Stack<T>();
    private readonly T _prefab;
    private readonly Transform _parent;

    public UnityPool(T prefab, Transform parent, int prewarm = 0)
    {
        _prefab = prefab;
        _parent = parent;
        for (int i = 0; i < prewarm; i++)
            _pool.Push(CreateOne());
    }

    private T CreateOne()
    {
        var obj = UnityEngine.Object.Instantiate(_prefab, _parent);
        obj.gameObject.SetActive(false);
        return obj;
    }

    public T Get()
    {
        var item = _pool.Count > 0 ? _pool.Pop() : CreateOne();
        item.OnSpawned();
        return item;
    }

    public void Return(T item)
    {
        item.OnDespawned();
        _pool.Push(item);
    }
}

// 使用
var pool = new UnityPool<EffectParticle>(effectPrefab, poolRoot, prewarm: 10);
var e = pool.Get();
pool.Return(e);
```

### 关键知识点

| 问题 | 答案 |
|------|------|
| 需要继承基类吗？ | **不需要**，用泛型约束即可 |
| 数据清洗放哪？ | 归还时（Return 时）调用，Reset 所有运行时状态 |
| 预热数量多少合适？ | 启动时常用的数量，减少运行时首次分配的卡顿 |
| 容量上限怎么定？ | 根据压测数据设定上限，超出时拒绝获取或创建临时对象 |
| 线程安全吗？ | 以上实现非线程安全，需要加 lock 或用 ConcurrentBag |

### 面试应答模板

> "通用内存池不需要继承特殊类。通过泛型约束 `where T : class` 或 `IPoolable` 接口约束具体类型。归还时通过 `Action<T>` 回调或 `IPoolable.OnDespawned()` 接口执行数据清洗。重置位置、清零状态、停止特效等都在这里做。我的设计还包括预热机制和容量上限保护，避免运行时动态分配。"

## 盲点 3：帧同步 vs 状态同步概念 🔴 高优先级

### 核心问题

你把"乐观预测 + 服务器校验"方案归类为帧同步，但严格来说这是两种不同的同步方案。

### 扫盲：四种网络同步模型

```
模型一：严格帧同步（LockStep）
┌──────────┐      输入帧       ┌──────────┐      输入帧       ┌──────────┐
│ Client A │ ─────────────────→│  Server  │ ←─────────────────│ Client B │
│ 跑逻辑   │                  │  只转发   │                  │ 跑逻辑    │
│ 帧0等A和B │                 │  广播输入  │                  │ 帧0等A和B  │
│ 的输入   │                  │           │                  │ 的输入    │
└──────────┘                  └──────────┘                  └──────────┘
→ 确定性：相同输入 → 相同结果 → 严格一致
→ 缺点：一个人卡，所有人都等；逻辑全在客户端，外挂成本低


模型二：乐观预测 + 服务器校验（你实际用的）
┌──────────┐                    ┌──────────┐
│ Client    │ ──操作→ Server    │
│ 本地预表现 │ ←─校验结果──     │ 权威逻辑  │
│ 立刻响应   │  不一致就回退     │ 每帧验算  │
└──────────┘                    └──────────┘
→ 玩家体验好（无感知延迟）
→ 服务器校验，安全


模型三：服务器权威状态同步
┌──────────┐                    ┌──────────┐
│ Client    │ ←─状态快照──     │  Server  │
│ 只渲染    │  ──操作→         │ 跑逻辑    │
│           │                   │ 每帧广播  │
└──────────┘                    └──────────┘
→ 逻辑全在服务器，最安全
→ 缺点：延迟明显


模型四：预测 + 插值（适合射击游戏）
┌──────────┐
│ Client    │  本地预测移动（立即响应）
│           │  收到服务器位置后平滑插值到正确位置
│           │  敌人位置用插值显示，不做预测
└──────────┘
→ 本地玩家无延迟感，远程玩家平滑
```

### 帧同步必须满足的条件

| 条件 | 说明 |
|------|------|
| 确定性 | 相同输入 + 相同状态 → 相同结果 |
| 浮点数禁用 | 必须用整数运算，或用 `DeterministicFloat` 库 |
| 统一随机种子 | 服务器开局下发 seed，本地所有随机数走同一算法 |
| 帧锁定 | 所有客户端在同一帧号执行，不提前不落后 |

### 面试应答模板

> "帧同步和状态同步的核心区别是'谁跑逻辑'。帧同步：所有客户端跑相同逻辑，服务器只广播输入（确定性保证）。状态同步：服务器跑逻辑，广播状态给客户端。
>
> 我们项目用的是**乐观预测 + 服务器校验**：客户端本地立即执行（玩家感觉无延迟），同时把操作发给服务器，服务器验算后返回结果。如果不一致，客户端回退到服务器状态。这兼顾了体验和安全性。帧同步更适合强竞技（MOBA、棋牌），状态同步适合 RPG、FPS。"

### 核心问题

### 扫盲：BFS 完整实现

```csharp
// BFS — 广度优先搜索（最短路径）
// 关键词：队列（先进先出）、逐层扩展

public class BFSSolver
{
    // 四方向移动
    private static readonly Vector2Int[] DIRS = new[]
    {
        new Vector2Int(0, 1), new Vector2Int(0, -1),
        new Vector2Int(1, 0), new Vector2Int(-1, 0)
    };

    // 求最短路径
    public List<Vector2Int> ShortestPath(Vector2Int start, Vector2Int end,
                                          Func<Vector2Int, bool> isValid)
    {
        var queue = new Queue<Vector2Int>();
        var visited = new HashSet<Vector2Int>();
        var parent = new Dictionary<Vector2Int, Vector2Int>();

        queue.Enqueue(start);
        visited.Add(start);

        while (queue.Count > 0)
        {
            var current = queue.Dequeue();

            if (current == end)
                return ReconstructPath(parent, start, end);

            foreach (var dir in DIRS)
            {
                var next = current + dir;
                if (isValid(next) && !visited.Contains(next))
                {
                    visited.Add(next);
                    parent[next] = current; // 记录来源
                    queue.Enqueue(next);
                }
            }
        }

        return null; // 无可达路径
    }

    // 反向追溯路径
    private List<Vector2Int> ReconstructPath(
        Dictionary<Vector2Int, Vector2Int> parent,
        Vector2Int start, Vector2Int end)
    {
        var path = new List<Vector2Int>();
        var curr = end;
        while (curr != start)
        {
            path.Add(curr);
            curr = parent[curr];
        }
        path.Add(start);
        path.Reverse(); // 反转为正向
        return path;
    }
}
```

public int OrangesRotting(int[][] grid)
{
    var queue = new Queue<(int, int, int)>(); // (r, c, time)
    int fresh = 0;

    // 初始化：所有腐烂橘子入队

        for (int j = 0; j < grid[0].Length; j++)
            if (grid[i][j] == 2) queue.Enqueue((i, j, 0));
            else if (grid[i][j] == 1) fresh++;

    int minutes = 0;
    while (queue.Count > 0)
    {
        var (r, c, t) = queue.Dequeue();
        minutes = Math.Max(minutes, t);

        foreach (var (nr, nc) in GetNeighbors4(r, c, grid))
        {
            if (grid[nr][nc] == 1)
            {
                grid[nr][nc] = 2;
                fresh--;
                queue.Enqueue((nr, nc, t + 1));
            }
        }
    }

### DFS 完整实现（递归 + 回溯）

```csharp
// DFS — 深度优先搜索
// 关键词：递归（或栈）、一条路走到黑、不行就回退

// 递归版（简洁）
public bool DFSSearch(Vector2Int pos, Vector2Int target,
                      Func<Vector2Int, bool> isValid,
                      HashSet<Vector2Int> visited)
{
    if (pos == target) return true;

    foreach (var dir in DIRS)
    {
        var next = pos + dir;
        if (isValid(next) && !visited.Contains(next))
        {
            visited.Add(next);
            if (DFSSearch(next, target, isValid, visited))
                return true;
            visited.Remove(next); // ← 关键：回溯
        }
    }
    return false;
}

// 栈版（避免递归爆栈）
public bool DFS_Iterative(Vector2Int start, Vector2Int target,
                          Func<Vector2Int, bool> isValid)
{
    var stack = new Stack<Vector2Int>();
    var visited = new HashSet<Vector2Int>();

    stack.Push(start);
    visited.Add(start);

    while (stack.Count > 0)
    {
        var current = stack.Pop();

        if (current == target) return true;

        foreach (var dir in DIRS)
        {
            var next = current + dir;
            if (isValid(next) && !visited.Contains(next))
            {
                visited.Add(next);
                stack.Push(next);
            }
        }
    }
    return false;
}
```

### BFS vs DFS 选择指南

| 场景 | 推荐 | 原因 |
|------|------|------|
| 最短路径（格子、图） | **BFS** | 按层扩展，第一层到达的就是最短 |
| 连通分量、岛屿数量 | **DFS/BFS** | 都可以 |
| 迷宫是否有解 | **DFS** | 只要找一个解，不需要最短 |
| 深度限制的搜索 | **DFS** | BFS 队列可能爆 |
| 拓扑排序 | **DFS** | 用 DFS 的返回顺序 |

## 盲点 5：算法题 — 二进制字符串相加 🟡 中优先级

### 核心问题

LeetCode 67，未完成。问题在：坐标偏移（数组越界）和 char → int 的类型转换。

### 详细解析与完整答案

```csharp
// LeetCode 67: Add Binary
// 输入："11" + "1" → "100"
// 输入："1010" + "1011" → "10101"

public string AddBinary(string a, string b)
{
    // 从字符串末尾开始，逐位相加，处理进位
    int i = a.Length - 1;       // ← a的最后一个字符（最低位）
    int j = b.Length - 1;       // ← b的最后一个字符（最低位）
    int carry = 0;              // 进位：0 或 1
    var sb = new System.Text.StringBuilder();

    // 关键：只要还有没处理的位，或者还有进位，就继续
    while (i >= 0 || j >= 0 || carry > 0)
    {
        int sum = carry;

        // 注意：char转int要用 -'0'，不能直接 (int)char
        // '0' = 48, '1' = 49, 所以 '1' - '0' = 1
        if (i >= 0) sum += a[i--] - '0';  // ← -'0' 是关键！
        if (j >= 0) sum += b[j--] - '0';

        sb.Append(sum % 2);   // 当前位：0 或 1
        carry = sum / 2;      // 进位：0 或 1
    }

    // 反转：因为是从低位到高位append的
    char[] arr = sb.ToString().ToCharArray();
    Array.Reverse(arr);
    return new string(arr);
}
```

### 你的 bug 分析

```
面试时的bug：
1. 坐标偏移：遍历时 i 和 j 的范围判断有误，导致数组越界或漏处理
2. char转int：直接 (int)a[i] 而不是 a[i] - '0'
   直接转换 '1' = 49, '0' = 48, 会把ASCII码值当作数字加

正确做法：
  int digit = a[i] - '0';  // '1' - '0' = 1, '0' - '0' = 0
```

### 手把手图解

```
  a = "1010" (10)
  b =  "101" (5)

  从右往左：
  i=3 j=2 sum=0+0+0=0 → '0' 进位0
  i=2 j=1 sum=0+1+0=1 → '1' 进位0
  i=1 j=0 sum=1+0+0=1 → '1' 进位0
  i=0 j=-1 sum=1+0+0=1 → '1' 进位0

  sb = "0111" → Reverse → "1110" (14 = 10 + 5) ✓
```

public int[] AddBinaryArrays(int[] a, int[] b) // a,b 元素是 0 或 1
{
    int carry = 0;
    var result = new List<int>();

    int i = a.Length - 1, j = b.Length - 1;

    {
        int sum = carry;
        if (i >= 0) sum += a[i--];

        carry = sum / 2;

## 盲点 6：设计模式 — 接口抽象能力 🟡 中优先级

### 核心问题

设计模式知道用，但接口抽象能力较弱。追问"需要继承什么"时答不上来。

### 扫盲：设计模式的正确用法

```csharp
// 发布-订阅模式的正确实现（你应该能写出来）

// 第一步：定义事件
public interface IEventContext { } // 标记接口

public class DamageEvent : IEventContext
{
    public int Damage;
    public GameObject Attacker;
    public GameObject Target;
}

// 第二步：定义订阅者接口
public interface IEventHandler<T> where T : IEventContext
{
    bool CanHandle(T context);  // 二次过滤
    void Handle(T context);
}

// 第三步：事件总线
public class EventBus
{
    private static readonly Dictionary<Type, List<Delegate>> _handlers =
        new Dictionary<Type, List<Delegate>>();

    public static void Subscribe<T>(IEventHandler<T> handler) where T : IEventContext
    {
        var type = typeof(T);
        if (!_handlers.ContainsKey(type))
            _handlers[type] = new List<Delegate>();
        _handlers[type].Add(handler);
    }

    public static void Unsubscribe<T>(IEventHandler<T> handler) where T : IEventContext
    {
        var type = typeof(T);
        if (_handlers.ContainsKey(type))
            _handlers[type].Remove(handler);
    }

    public static void Publish<T>(T evt) where T : IEventContext
    {
        var type = typeof(T);
        if (_handlers.TryGetValue(type, out var list))
        {
            foreach (var handler in list.ToArray()) // ToArray防并发问题
            {
                var h = (IEventHandler<T>)handler;
                if (h.CanHandle(evt))
                    h.Handle(evt);
            }
        }
    }
}

// 第四步：具体订阅者
public class HealthBar : MonoBehaviour, IEventHandler<DamageEvent>
{
    public bool CanHandle(DamageEvent ctx) =>
        ctx.Target == gameObject; // 只响应自己受到的伤害

    public void Handle(DamageEvent ctx)
    {
        // 更新血条
    }
}

// 使用
EventBus.Subscribe(myHealthBar);
EventBus.Publish(new DamageEvent { Damage = 100, Target = enemy });
```

### 工厂模式

```csharp
// 工厂模式：创建有共同接口的对象
public interface IEffect { void Play(); void Stop(); }

public class FireEffect : IEffect { ... }
public class IceEffect : IEffect { ... }
public class LightningEffect : IEffect { ... }

public enum EffectType { Fire, Ice, Lightning }

public class EffectFactory
{
    private readonly Dictionary<EffectType, Func<IEffect>> _creators =
        new Dictionary<EffectType, Func<IEffect>>();

    public EffectFactory()
    {
        _creators[EffectType.Fire] = () => new FireEffect();
        _creators[EffectType.Ice] = () => new IceEffect();
        _creators[EffectType.Lightning] = () => new LightningEffect();
    }

    public IEffect Create(EffectType type)
    {
        if (_creators.TryGetValue(type, out var creator))
            return creator();
        throw new ArgumentException($"Unknown effect type: {type}");
    }
}
```

### 面试应答模板

> "设计模式的核心目的是**解耦**和**复用**。
>
> 发布-订阅模式：发布者和订阅者不直接依赖，通过事件总线解耦。新增订阅者不影响发布者。工厂模式：将对象创建集中管理，隐藏具体类型，修改创建逻辑只需改一处。单例模式：全局唯一实例，但要注意线程安全（用 double-checked locking）和生命周期管理。策略模式：不同算法可替换，例如伤害计算可以用物理策略、魔法策略。"

## 盲点 7：Dependency Injection（依赖注入） 🔵 中低优先级

### 核心问题

面试中没有涉及，但 Flexi 框架源码中大量使用 DI，是你简历上"战斗系统"方向的必备知识。

### 扫盲

```csharp
// 依赖注入：高层模块不依赖低层模块，都依赖抽象

// 场景：ResourceManager 依赖 ISerializer

// 反面（紧耦合）：
public class ResourceManager
{
    private BinarySerializer _serializer = new BinarySerializer(); // 直接依赖实现
}

// 正面（依赖注入）：
public interface ISerializer
{
    byte[] Serialize<T>(T obj);
    T Deserialize<T>(byte[] data);
}

public class BinarySerializer : ISerializer { ... }
public class JsonSerializer : ISerializer { ... }

// 在 ResourceManager 中注入
public class ResourceManager
{
    private readonly ISerializer _serializer;

    // 通过构造函数注入（最常见方式）
    public ResourceManager(ISerializer serializer)
    {
        _serializer = serializer;
    }
}

// 在 Flexi 中的体现：
var builder = new FlexiCoreBuilder();
builder.SetEventResolver(myEventResolver);       // 注入接口实现
builder.SetStatRefreshResolver(myStatResolver); // 注入接口实现
var core = builder.Build();
```

### 优势

- **可测试性**：测试时替换为 Mock 实现
- **可替换性**：换序列化方案只需改注入的实例
- **解耦**：高层不依赖具体低层实现

## 盲点 8：事件风暴与性能 🟡 中优先级

### 核心问题

你知道用事件总线，但没提到事件风暴的风险和解决方案。

### 扫盲

```csharp
// 问题：大量事件触发大量订阅者 → 性能爆炸
// 例如：OnHealthChanged 事件 → 触发100个订阅者 → 每帧多次触发

// 解决方案一：分层事件
public static class EventChannels
{
    public static readonly string UI_Health = "ui.health";       // UI层
    public static readonly string Game_Damage = "game.damage";   // 游戏逻辑层
    public static readonly string System_Perf = "system.perf";   // 系统层
    // 按模块/层级分离，避免跨层干扰
}

// 解决方案二：事件合并（同一帧内多次触发只执行一次）
public class ThrottledEvent<T>
{
    private readonly List<Action<T>> _handlers = new List<Action<T>>();
    private T _pendingData;
    private bool _pending = false;

    public void Publish(T data)
    {
        _pendingData = data;
        _pending = true; // 只标记，不立即分发
    }

    public void Flush()
    {
        if (_pending)
        {
            foreach (var h in _handlers) h(_pendingData);
            _pending = false;
        }
    }
}

// 解决方案三：CanHandle 过滤
// 每个订阅者在响应前做一次检查，不符合条件的直接跳过
public interface IEventFilter<T>
{
    bool ShouldHandle(T evt);
}
```
