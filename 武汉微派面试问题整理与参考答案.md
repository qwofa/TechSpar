### Q1：你用过哪些设计模式？

**你的回答：** 观察者/发布订阅、工厂模式。

**追问：什么时候用订阅模式？**

**你的回答：** 棋盘状态变化时通知棋子，每个棋子检查自己是否需要响应。

**资深标准回答：**

发布-订阅模式的核心场景：

```csharp
// 典型场景一：UI 与逻辑解耦
public class EventBus
{
    private static readonly Dictionary<string, List<Delegate>> _handlers = new Dictionary<string, List<Delegate>>();

    public static void Subscribe<T>(string eventId, Action<T> handler) { ... }
    public static void Unsubscribe<T>(string eventId, Action<T> handler) { ... }
    public static void Publish<T>(string eventId, T data) { ... }
}

// 典型场景二：模块间通信（棋盘 ↔ 棋子）
public interface IEventHandler
{
    void Handle(IEventContext context);
}

// 使用方式
_board.OnStateChanged += (state) =>
{
    foreach (var piece in _pieces)
        piece.OnBoardStateChanged(state);
};
```

**回答要点：**
- **什么时候用**：一对多依赖、跨模块通信、事件驱动的流程。用发布-订阅让发布者和订阅者解耦，发布者不需要知道谁在监听。
- **对比直接调用**：直接调用简单但耦合紧，新增订阅者必须改发布者代码。发布-订阅增加了灵活性，代价是调试困难（事件链路不直观）。
- **注意事项**：避免事件风暴（大量事件触发导致性能问题），建议配合 `IHandle<T>.CanHandle()` 做过滤。

### Q2：讲讲你的"真同步"设计？为什么用帧同步？

**你的回答：** 玩家输入不能因网络波动而延迟，本地预表现 + 服务器仲裁。单人玩法也走服务器是为了游戏资产安全（奖励生成、随机数种子统一）。

**追问：单机玩法为什么不用纯本地运算？**

**你的回答：** 因为奖励是服务器权威生成的，货币、道具不能前端自己加。

**资深标准回答：**

**帧同步（LockStep）vs 状态同步（State Sync）对比：**

| 维度 | 帧同步 | 状态同步 |
|------|--------|----------|
| 核心原理 | 所有客户端执行相同逻辑，服务器只广播输入 | 服务器权威模拟，广播状态快照 |
| 一致性 | 严格一致（确定性） | 依赖网络，可能抖动 |
| 延迟感知 | 玩家立即看到自己操作 | 需要等服务器确认 |
| 反外挂 | 弱（逻辑在客户端） | 强（逻辑在服务器） |
| 适用场景 | 强竞技、MOBA、棋牌 | MMORPG、FPS、弱竞技 |

**你实际用的方案（从面试描述判断）：**
> 这更像是"乐观预测 + 服务器校验"而非严格帧同步。严格帧同步中服务器不会做"校验"和"回退"——帧同步中服务器和客户端跑的是**同一份代码**，每帧输入一致，结果必然一致。服务器只是按固定帧率广播输入。

**资深程序员会这样说：**

> "我们项目用的是帧同步方案，核心设计如下：
>
> 1. **确定性保证**：所有计算使用整数（Int32/Int64），避免浮点数。随机数使用统一的种子（开局时服务器下发）。
> 2. **输入录制**：每个玩家操作在当前帧生成一个 `InputFrame`（包含操作类型、目标、参数），广播给所有客户端。
> 3. **帧同步循环**：每个客户端在帧 N 的所有输入收齐后，才推进到帧 N+1。用 `WaitForFixedUpdate()` 保证固定帧率。
> 4. **断线重连**：服务器保存最近 N 帧的输入序列，重连时下发补帧数据，客户端补算到当前帧。
> 5. **服务器校验**：服务器也跑一份逻辑，如果与客户端结果不一致，说明有外挂或 bug，记录日志并告警。"
>
> 如果面试官追问你"单人玩法为什么还走服务器"，更准确的回答是：
> - 如果只是纯本地运算，可以不走服务器
> - 但只要涉及**资产/奖励/随机数**，就必须服务器权威
> - 更好的设计是：纯玩法逻辑走帧同步，奖励系统走独立的服务器请求，两者分离

### Q3：值类型和引用类型的区别？GC相关？

**你的回答：** 值类型存栈，引用类型存堆，堆由 GC 释放，频繁拆箱装箱影响性能，内存池可复用。

**资深标准回答：**

```csharp
// 值类型：struct，直接存储在栈或内联于引用类型的内部
struct Point { public int X; public int Y; }

// 引用类型：class，存在堆上，通过栈上的引用访问
class Player { public int Health; public string Name; }

// 关键点一：存储位置取决于"容器"，不是绝对的
class Example
{
    Point p1;              // 值类型作为class成员 → 存在堆上（随对象一起）
    Point[] arr;           // 数组在堆上，每个元素在堆上
    int local = 5;         // 局部变量 → 在栈上
}

// 关键点二：装箱拆箱
int a = 5;
object o = a;     // 装箱：值类型→堆上new object，触发分配
int b = (int)o;   // 拆箱：访问堆上数据，可能触发GC

// 关键点三：GC的影响
// GC只管堆，栈自动释放
// GC停顿（Stop The World）影响性能
// 频繁分配短生命周期对象 → 大量GC → 卡顿
// 解决方案：对象池、栈分配(Span<T>)、结构体、ArrayPool<T>
```

**回答要点：**
- 存储位置：**栈**存局部变量和值类型本身；**堆**存引用类型对象和作为类成员的值类型
- GC 只管堆，栈自动管理
- 避免 GC 的方法：对象池、内存池、`ArrayPool<T>`、`Span<T>`、值类型替代小对象

### Q4：通用内存池怎么设计？

**你的回答：** 提供获取/归还接口，回收时数据清洗，容量不足时扩容。

**追问：通用内存池需要继承什么？数据清洗怎么做？**

**你的回答：** （答不上来，承认只用过类型确定的内存池）

**资深标准回答：**

```csharp
// 方案一：泛型内存池（推荐，最常用）
public class ObjectPool<T> where T : class, new()
{
    private readonly Stack<T> _pool = new Stack<T>();
    private readonly Func<T> _factory;
    private readonly Action<T> _onReset;  // 数据清洗回调

    public ObjectPool(Func<T> factory, Action<T> onReset = null, int prewarm = 0)
    {
        _factory = factory;
        _onReset = onReset;
        for (int i = 0; i < prewarm; i++)
            _pool.Push(_factory());
    }

    public T Get()
    {
        return _pool.Count > 0 ? _pool.Pop() : _factory();
    }

    public void Return(T obj)
    {
        _onReset?.Invoke(obj);  // 清洗数据
        _pool.Push(obj);
    }
}

// 使用泛型接口，不需要继承特殊类
public interface IPoolable
{
    void OnSpawn();   // 被获取时调用
    void OnDespawn(); // 被归还时调用 ← 数据清洗在这里
}

public class Bullet : IPoolable
{
    public void OnSpawn() { }
    public void OnDespawn()
    {
        // 清洗：重置位置、速度、状态等
        Velocity = Vector3.zero;
        Damage = 0;
        Target = null;
    }
}

// 使用
var bulletPool = new ObjectPool<Bullet>(() => new Bullet(),
    b => b.OnDespawn(), prewarm: 20);
var bullet = bulletPool.Get();
bulletPool.Return(bullet);
```

```csharp
// 方案二：接口型内存池（更通用）
public interface IResettable
{
    void Reset();
}

// 通用池子，约束为 IResettable
public class GenericPool<T> where T : class, IResettable, new()
{
    private readonly Stack<T> _available = new Stack<T>();

    public T Rent() => _available.Count > 0 ? _available.Pop() : new T();
    public void Return(T item) { item.Reset(); _available.Push(item); }
}

// 使用示例
public class Effect : IResettable
{
    public float Duration;
    public ParticleSystem Particle;
    public void Reset() { Duration = 0; Particle?.Stop(); }
}
```

**回答要点：**
- **不需要继承基类**，通过泛型约束或接口实现
- 数据清洗用回调 `Action<T>` 或 `IResettable.Reset()` 接口
- 扩容策略：按需扩容、预热、容量上限保护
- Unity 中 UnityEngine.Object 的池化需要特殊处理（`Object.DontDestroyOnLoad` 等）

### Q5：Unity 内置类型（贴图、GameObject）的内存结构？

**你的回答：** 都是引用类型，贴图和 material 可以共用内存，通过引用计数管理。不太清楚贴图具体占哪块内存。

**追问：贴图在内存中占哪块区域？**

**你的回答：** 不太确定，应该在托管堆。

**资深标准回答：**

Unity 内存分为三大区域：

| 区域 | 内容 | 特点 |
|------|------|------|
| **托管堆（Managed Heap）** | C# 对象、字符串、数组、List、Dictionary | GC 管理，可自动扩容 |
| **原生堆（Native Heap）** | 贴图、Mesh、Shader、AudioClip、Prefab | Unity 底层管理，不受 GC 控制 |
| **栈（Stack）** | 值类型局部变量、方法调用参数、返回地址 | 线程独有，自动释放 |

**贴图的内存结构：**

```csharp
// 贴图内存占用计算
// 贴图内存(bytes) = 宽 × 高 × 每像素字节数 × MipMap层级数系数

// 例如：2048×2048 RGBA32 贴图
// = 2048 × 2048 × 4 bytes/pixel ≈ 16 MB

// Unity贴图格式对应内存大小：
// RGB24   → 3 bytes/pixel
// RGBA32  → 4 bytes/pixel
// RGBA64  → 8 bytes/pixel
// BC1/DXT1 → 0.5 bytes/pixel（压缩格式，质量低）
// BC5      → 1 bytes/pixel（法线图常用）
// ASTC    → 压缩比可变（4×4~12×12）

// MipMap 增加约 1/3 额外内存
// 所以 2048×2048 贴图 + MipMap ≈ 21 MB
```

**GameObject 内存：**
- GameObject 本身很小（约 2-3KB），主要开销在组件
- 每个 Component（Transform、MeshRenderer、Collider 等）占用额外内存
- 挂载的脚本（MonoBehaviour）在托管堆分配

**共用机制（Reference Counting）：**
```csharp
// 当两个物体使用同一材质/贴图时
// Resources.Load("MatA") → 引用计数=1
// Resources.Load("MatA") → 引用计数=2（返回同一实例）
// 卸载：引用计数-1，归零后标记为可卸载
// 注意：Resources.UnloadUnusedAssets() 或 GC.Collect() 才真正释放
```

**回答要点：**
- 贴图在**原生堆**，不在托管堆
- 托管堆只存 C# 脚本对象、字符串、集合
- 原生内存需要手动管理（`Resources.UnloadAsset()`、`AssetBundle.Unload(true)`）
- 混用两者会导致内存碎片化

### Q6：协程的原理？

**你的回答：** 生成器模式，闭包，保存执行位置指针，每帧移动到下一个 yield 处。

**资深标准回答：**

```csharp
// 协程的本质：C# 的 IEnumerator + yield return
// Unity 使用 C# 的迭代器块（Iterator Block）实现

// 这段代码...
IEnumerator LoadAsset()
{
    Debug.Log("开始加载");
    yield return null;           // 第1帧停这里
    var req = Resources.LoadAsync("big");
    yield return req;            // 等异步完成，停这里
    Debug.Log("加载完成");
    yield return new WaitForSeconds(2f); // 停2秒
    Debug.Log("2秒后");
}

// ...编译器会生成一个状态机类（约等于是这样的结构）
class LoadAssetEnumerator : IEnumerator<object>
{
    private int _state = 0;
    private DateTime _waitUntil;

    public bool MoveNext()
    {
        switch (_state)
        {
            case 0:
                Debug.Log("开始加载");
                _state = 1;
                return true;
            case 1:
                if (Time.time < _waitUntil) return true; // 继续等待
                _state = 2;
                return false;
        }
        return false;
    }
}

// Unity 的调度：每帧在 LateUpdate 驱动所有 StartCoroutine 的 MoveNext()
```

**Unity 协程调度流程：**
```
MonoBehaviour.StartCoroutine(IEnumerator)
  → 创建协程对象，加入协程调度链表
  → 每帧 LateUpdate 遍历链表，调用每个协程的 MoveNext()
  → MoveNext() 返回 false → 协程结束，从链表移除
```

**yield return 的类型支持：**
```csharp
yield return null;                    // 等待一帧
yield return new WaitForFixedUpdate(); // 等待物理帧
yield return new WaitForSeconds(2f);   // 等待2秒（受 Time.timeScale 影响）
yield return new WaitForSecondsRealtime(2f); // 不受 timeScale 影响
yield return new WaitUntil(() => flag); // 等待条件为真
yield return new WaitWhile(() => flag); // 等待条件为假
yield return www;                      // 等待异步请求完成
```

**协程 vs 多线程：**
- 协程是**伪并发**（假 concurrency），跑在主线程，只是不阻塞
- 多线程真并发，可利用多核 CPU
- Unity 主线程限制：所有 Unity API（Transform、GameObject、Physics）只能在主线程调用
- 耗时计算（AI寻路、大数据解压）用 Job System 或多线程

**协程的坑：**
- 协程开启后，即使脚本 Disabled 也会继续执行（除非手动 StopCoroutine）
- StartCoroutine 返回的 Coroutine 对象可以 Stop
- 不要在协程里无限循环 `while(true)` 而不 yield，否则卡死
- 协程无法返回值，用回调或 `Task<T>` 替代

### Q7：深度优先（DFS）和广度优先（BFS）算法？

**你的回答：** DFS 递归最深层再回溯，适合小数据；BFS 用队列，适合大数据、图问题。

**追问：BFS 怎么用代码实现？**

**你的回答：** 用队列。从起点入队，依次出队并把四个方向邻接点入队。用一个标记数组避免重复访问。

**资深标准回答：**

```csharp
// BFS — 广度优先搜索（最短路径）
public List<Vector2Int> BFS(Vector2Int start, Vector2Int target, Grid grid)
{
    var queue = new Queue<Vector2Int>();
    var visited = new HashSet<Vector2Int>();
    var parent = new Dictionary<Vector2Int, Vector2Int>();

    queue.Enqueue(start);
    visited.Add(start);

    while (queue.Count > 0)
    {
        var current = queue.Dequeue();

        if (current == target)
            return ReconstructPath(parent, start, target);

        foreach (var dir in new[] {
            new Vector2Int(0,1), new Vector2Int(0,-1),
            new Vector2Int(1,0), new Vector2Int(-1,0) })
        {
            var next = current + dir;
            if (grid.IsValid(next) && !visited.Contains(next))
            {
                visited.Add(next);
                parent[next] = current;
                queue.Enqueue(next);
            }
        }
    }
    return null; // 无路径
}

// DFS — 深度优先搜索（回溯）
public bool DFS(Vector2Int pos, Vector2Int target, Grid grid,
                 HashSet<Vector2Int> visited)
{
    if (pos == target) return true;

    foreach (var dir in new[] {
        new Vector2Int(0,1), new Vector2Int(0,-1),
        new Vector2Int(1,0), new Vector2Int(-1,0) })
    {
        var next = pos + dir;
        if (grid.IsValid(next) && !visited.Contains(next))
        {
            visited.Add(next);
            if (DFS(next, target, grid, visited))
                return true;
            visited.Remove(next); // 回溯
        }
    }
    return false;
}

// 游戏中的实际应用：
// BFS：寻路（战棋游戏的格子移动范围）、迷雾展开、连通分量
// DFS：迷宫生成、递归关卡解锁、树/图遍历
```

**两种算法的对比：**

| 维度 | BFS（队列） | DFS（栈/递归） |
|------|-------------|----------------|
| 空间复杂度 | O(N)（最坏队列存所有节点） | O(H)（递归栈深度=树高） |
| 时间复杂度 | O(V+E) | O(V+E) |
| 最短路径 | ✅ 能保证（每层逐步扩展） | ❌ 不保证 |
| 适合场景 | 最短路径、层级遍历、网格扩展 | 连通性检测、拓扑排序、回溯 |
| 递归深度 | 不递归 | 递归有爆栈风险（H过大时） |

### Q8：资源加载管理系统重构 — 怎么做的？

**你的回答：** 解决了两个问题：1）代码重复缓存路径和资源；2）卸载依赖手动管理。改为弱引用追踪，模块关闭时自动降引用。

**追问：引用归零后立即释放吗？分帧释放怎么做？**

**你的回答：** 不确定是否立即释放。分帧是为了控制每帧释放数量，避免卡顿。

**资深标准回答：**

```csharp
public class ResourceReference

    }

// 3. 分帧释放策略（避免 GC 卡顿）
public class ResourceUnloadScheduler
{
    private Queue<UnityEngine.Object> _unloadQueue = new Queue<UnityEngine.Object>();
    private const int MAX_UNLOAD_PER_FRAME = 5;
    private const int MAX_UNLOAD_TIME_MS = 2; // 每帧最多花2ms卸载

    public void EnqueueUnload(UnityEngine.Object asset)
    {
        _unloadQueue.Enqueue(asset);
    }

    public void ProcessUnloads()
    {
        int unloadedThisFrame = 0;
        var sw = System.Diagnostics.Stopwatch.StartNew();

        while (_unloadQueue.Count > 0 &&
               unloadedThisFrame < MAX_UNLOAD_PER_FRAME &&
               sw.ElapsedMilliseconds < MAX_UNLOAD_TIME_MS)
        {
            var asset = _unloadQueue.Dequeue();
            Resources.UnloadAsset(asset);
            unloadedThisFrame++;
        }

        if (_unloadQueue.Count == 0)
            Resources.UnloadUnusedAssets(); // 清理未使用资源
    }
}

// 4. UI 层级缓存策略（腾讯方案简化版）
public enum UILayer { FullScreen, Popup, Tips }

public class UIPoolManager
{
    private Dictionary<UILayer, Stack<UIPanel>> _pools = new Dictionary<UILayer, Stack<UIPanel>>();
    private const int MAX_FULLSCREEN = 3; // 最多3个全屏界面缓存

    public UIPanel Get(UILayer layer)
    {
        if (_pools[layer].Count > 0)
            return _pools[layer].Pop();
        return CreateNewPanel(layer);
    }

    public void Return(UIPanel panel, UILayer layer)
    {
        if (layer == UILayer.FullScreen && _pools[layer].Count >= MAX_FULLSCREEN)
        {
            panel.DestroyImmediate(); // 超出容量直接销毁
            return;
        }
        _pools[layer].Push(panel);
    }
}
```

**回答要点：**
- 引用归零后，Unity 不会立即释放（GC 的 Stop The World 会分批处理）
- 分帧释放：控制每帧释放数量 + 限制每帧耗时，避免卡顿
- 自动释放：弱引用追踪模块生命周期，模块销毁时自动降引用 → 触发释放

### Q9：UI 界面部分资源卸载了、另一部分没卸载，加载新界面会出问题吗？

**你的回答：** 只要保证先卸载完成、再加载，就不会有问题。

**资深标准回答：**

这个回答**不够严谨**。实际场景有隐患：

```csharp
// 问题场景：
// 旧界面UI引用了 TextureA + TextureB
// 卸载时 TextureA 被释放了，TextureB 因其他模块引用还在
// 此时加载新界面，新界面也引用 TextureA
// 结果：TextureA 可能正在被重新加载，引用计数冲突

// 解决方案一：同步等待卸载完成
public IEnumerator LoadUIWithSafeUnload(string uiPath)
{
    // 1. 先卸载旧界面
    UnloadCurrentUI();

    // 2. 等待所有资源真正释放（引用归零 + 下一帧确认）
    yield return null; // 让 GC 有机会运行
    yield return Resources.UnloadUnusedAssets();

    // 3. 再加载新界面
    var handle = Resources.LoadAsync<GameObject>(uiPath);
    yield return handle;
    Instantiate(handle.asset);
}

// 解决方案二：引用计数 + 版本号保护
public class SafeResourceLoader
{
    private Dictionary<string, int> _loadingVersions = new Dictionary<string, int>();

    public AsyncOperation LoadWithGuard(string path, System.Action<UnityEngine.Object> onLoaded)
    {
        var version = ++_loadingVersions[path];
        var handle = Resources.LoadAsync<GameObject>(path);
        handle.completed += op =>
        {
            // 检查是否是最新的加载请求（避免被旧请求覆盖）
            if (_loadingVersions[path] == version)
                onLoaded?.Invoke((op as ResourceRequest).asset);
        };
        return handle;
    }
}
```

**标准回答：**
> "不会有问题，但需要保证时序正确。我的设计是：卸载和加载之间有状态机保证互斥——只有当卸载流程完全结束后，才允许触发加载流程。卸载过程中产生的资源引用变化（部分释放）不会影响加载，因为加载的资源路径是固定的，引用计数在加载时重新建立。"

### Q11：对 GAS（Gameplay Ability System）框架的理解？

**你的回答：** 技能作为资产实例化数据，数据驱动，属性分基础层和修饰符层，通过事件订阅触发，用状态机驱动切换。

**资深标准回答：**

> "GAS 的核心理念是**数据驱动 + 组件化**。
>
> 1. **Ability as Asset**：每个技能是一个 ScriptableObject，设计师可以拖拽、配置参数，不改代码就能新增技能
> 2. **Attribute System**：属性分 BaseValue 和 CurrentValue。Modifier（Add/Multiply/Override）分层叠加，避免每次属性变化都重算所有技能
> 3. **Gameplay Effect**：Buff/Debuff 封装为 GameplayEffect，可以堆叠（Stacking）、有持续时间、可配置清除条件
> 4. **Event-driven**：通过 GameplayEvent 触发 Ability，Ability 之间可以互相通信
> 5. **Execution Order**：Ability 执行顺序由 Tag、Priority 共同决定，避免技能乱序
>
> 相比传统硬编码技能的优势：设计师可独立调整技能参数，程序只需维护框架，新增技能成本极低。"

**Flexi（你简历旁边的速成指南里的框架）的核心要点：**
- FlexiCore 负责调度（Enqueue / Run / StatRefresh）
- EntryNode 是事件入口，ProcessNode 是执行逻辑
- ModifierNode 在 StatRefresh 时机统一应用修饰符（不是立即生效）
- IFlexiEventResolver 决定哪些 Ability 响应事件
- 关键设计：依赖注入（通过接口注入 EventResolver、StatRefreshResolver）

```csharp
// 方案一：全量同步（原有方案）

// 问题：数据量大时带宽爆炸

// 方案二：脏标记 + 增量同步（你用的）

        set

        return this; // 实际中需要深拷贝

// 方案五：状态同步（不是帧同步）
// 适合不需要严格一致性的场景
// 服务器权威模拟，客户端做预测和插值

// 选型依据：
// 1. 数据量：字段多 → 增量；字段少 → 全量也 OK
// 2. 实时性要求：要求高 → 高频；允许延迟 → 降频
```

```csharp
// Binary Diff 实现
public byte[] ComputeDiff(object oldData, object newData)
{
    // 1. 序列化两个对象
    var oldBytes = Serialize(oldData);
    var newBytes = Serialize(newData);

    // 2. 逐字段对比，标记变化字段
    var diff = new List<byte>();
    var fields = GetChangedFields(oldData, newData);
    foreach (var field in fields)
    {
        diff.AddRange(BitConverter.GetBytes(field.Id));
        diff.AddRange(field.NewValue);
    }

    // 3. 可选：压缩
    return Compress(diff.ToArray());
}

// Binary Patch 实现
public void ApplyPatch(object target, byte[] diff)
{
    using var ms = new MemoryStream(diff);
    using var br = new BinaryReader(ms);

    while (ms.Position < ms.Length)
    {
        int fieldId = br.ReadInt32();
        byte[] value = br.ReadBytes(br.ReadInt32());
        ApplyField(target, fieldId, value);
    }
}
```

**题目：** 给两个二进制字符串，返回它们的和（也是二进制字符串）。例如 `"11" + "1" = "100"`。

**标准答案（多种解法）：**

```csharp
// 解法一：位运算（最优）
public string AddBinary(string a, string b)
{
    int i = a.Length - 1, j = b.Length - 1, carry = 0;
    var result = new System.Text.StringBuilder();

    while (i >= 0 || j >= 0 || carry > 0)
    {
        int sum = carry;
        if (i >= 0) sum += a[i--] - '0';
        if (j >= 0) sum += b[j--] - '0';

        result.Append(sum % 2);  // 当前位
        carry = sum / 2;         // 进位
    }

    // 反转（因为是从低位到高构建的）
    var arr = result.ToString().ToCharArray();
    Array.Reverse(arr);
    return new string(arr);
}

// 解法二：转整数相加（适合短字符串，但有溢出风险）
public string AddBinary_V2(string a, string b)
{
    int intA = Convert.ToInt32(a, 2);
    int intB = Convert.ToInt32(b, 2);
    int sum = intA + intB;
    return Convert.ToString(sum, 2);
}

// 解法三：手动模拟竖式（最直观）
public string AddBinary_V3(string a, string b)
{
    // 对齐，短的前面补0
    int maxLen = Math.Max(a.Length, b.Length);
    a = a.PadLeft(maxLen, '0');
    b = b.PadLeft(maxLen, '0');

    int carry = 0;
    var result = new char[maxLen];

    for (int k = maxLen - 1; k >= 0; k--)
    {
        int bitA = a[k] - '0';
        int bitB = b[k] - '0';
        int total = bitA + bitB + carry;

        result[k] = (total % 2) == 1 ? '1' : '0';
        carry = total / 2;
    }

    if (carry > 0)
        return carry + new string(result);
    return new string(result);
}
```

**面试技巧：**
- 先和面试官确认输入范围（字符串最长多少？是否可能有空字符串？）
- 先说思路，再写代码。思路对了，代码有小 bug 也比写不出来好
- 位运算是最优解，时间 O(max(n,m))，空间 O(max(n,m))
