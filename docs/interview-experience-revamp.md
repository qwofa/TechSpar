# 面试体验系统分期改造方案

这份文档沉淀本轮改造路线，第一批只落 P0 + P1：先把抽象定型，再让简历模拟面试具备 4 档稳定体验。

## 总目标

TechSpar 后续不应继续在各页面各写一套面试逻辑，而应收敛成一套可复用的面试控制协议：

1. 先统一控制抽象。
2. 再提供固定挡位。
3. 再把挡位落到启动、恢复、重开、复盘这些真实链路。
4. 后续再扩展到行为量化、DIY 和多模式复用。

## 分期总览

| 阶段 | 目标 | 用户可见性 | 当前批次 |
| --- | --- | --- | --- |
| P0 规格定型 | 固定控制轴、行为预算、挡位、状态流和验收口径 | 部分不可见 | 本批执行 |
| P1 固定挡位改造 | 简历模拟面试开始前提供 4 档选择，并贯穿运行链路 | 可见 | 本批执行 |
| P2 行为量化与复盘闭环 | 让挡位差异能被复盘解释和回归验证 | 可见 | 后续 |
| P3 简历门禁 DIY | 基于简历预生成包开放有限定制控制台 | 强可见 | 后续 |
| P4 稳定性与验收体系 | 缓存、版本、统计、回归样例 | 弱可见 | 后续 |
| P5 扩模式复用 | 推广到 JD 备面、专项训练、Copilot | 可见 | 后续 |

## P0：规格定型

### 控制轴

| 控制轴 | 作用 |
| --- | --- |
| 面试官人格 | 决定面试官是教练型、标准型、专家型还是压力型 |
| 施压强度 | 决定追问是否紧、是否直接挑战候选人表述 |
| 验证方式 | 决定更偏基础澄清、项目真实性、横向方案还是取舍判断 |
| 节奏密度 | 决定问题推进速度和阶段切换速度 |
| 考察重心 | 决定技术、项目、岗位贴合、行为面的权重 |
| 上下文贴合 | 决定问题贴简历和目标岗位的程度 |

### 行为预算

| 预算项 | 含义 |
| --- | --- |
| 基础澄清 | 先确认概念、背景和候选人的真实表述 |
| 深度追问 | 沿着一个回答继续向底层机制、边界和细节追问 |
| 横向方案 | 让候选人比较替代方案或迁移到新场景 |
| 方案取舍 | 关注为什么这么做、代价是什么、有什么 trade-off |
| 优化扩展 | 关注性能、稳定性、规模化和可维护性 |
| 项目真实性验证 | 检查项目是否真做过、细节是否自洽 |
| 岗位贴合 | 对齐目标岗位能力模型和简历经历 |
| 行为验证 | 用具体经历验证协作、抗压、复盘和沟通能力 |

### 4 个固定挡位

| 挡位 | 面试体验 | 适合场景 |
| --- | --- | --- |
| 友好训练 | 更像教练，先帮用户把经历说清楚 | 第一次练、简历表达还没收口 |
| 标准面试 | 接近常规技术面，节奏和压力适中 | 建立基线、正式投递前自测 |
| 深挖验证 | 明显加大项目真实性和技术细节追问 | 项目经历强但担心被追问穿 |
| 高压实战 | 节奏更紧，更多挑战、反问和方案取舍 | 冲刺高强度面试或终面压力场 |

### P0 验收口径

P0 结束后，内部必须能稳定回答：

1. 同一个挡位为什么和其他挡位不一样。
2. 同一个行为预算项到底代表什么。
3. 启动、恢复、重开、复盘为什么可以沿用同一个控制对象。

## P1：固定挡位改造

### 本批范围

| 对象 | 本批做什么 | 暂不做什么 |
| --- | --- | --- |
| 简历模拟面试入口 | 开始前选择 4 档固定挡位 | 不开放高级 DIY |
| 后端启动协议 | 接收并校验挡位，写入会话元信息 | 不做实时跨档自适应 |
| 面试官运行时 | 将挡位转成面试官行为约束 | 不引入复杂预算统计 |
| 会话恢复 | 恢复页面能拿回同一挡位 | 不重建历史旧会话 |
| 再次练习 | 复用上一场目标岗位和挡位 | 不改变 JD 备面、专项训练体验 |
| 复盘返回 | 复盘页能拿到本场挡位信息 | 不做完整 P2 复盘解释 |

### 状态流

```text
简历入口
  -> 选择目标岗位 + 固定挡位
  -> /api/interview/start
  -> 后端校验挡位并写入 session.meta
  -> LangGraph 用同一控制对象生成面试官开场
  -> 面试中断后 /api/interview/session/:id/resume 恢复 meta
  -> 结束后 /api/interview/review/:id 返回 meta
  -> 复盘页再次练习复用 meta.target_role + meta.interview_control
```

### 复用边界

本批新增的复用对象分三层：

1. 后端 `interview_control`：控制轴、行为预算、4 档配置和会话 payload 补齐。
2. 前端 `interviewControl`：用于展示、兜底和从不同 payload 读取同一控制对象。
3. 前端 `useSessionLauncher`：统一会话启动的 loading、错误和跳转协议，避免页面各自维护一套状态。

## 后续阶段边界

P2 再做行为预算统计、复盘解释和挡位差异回归样例。

P3 再做简历门禁 DIY：用户上传简历后预生成预览包，DIY 只在固定挡位底座上做有限调整。

P4 再补缓存键、失效规则、协议版本升级和稳定性验收。

P5 再将同一控制协议推广到 JD 备面、专项训练和 Copilot。
