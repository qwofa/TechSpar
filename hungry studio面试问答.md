## 一、资源加载与内存优化（YooAsset）

### Q1：重构 YooAsset 资源加载管理系统时，解决了哪些核心问题？

**标准答案：**

**问题一：双重引用计数导致的内存泄漏。**

原有系统存在两套引用计数体系：YooAsset 内部维护一套，业务层的 `ResourceManager` 也维护一套引用。当卸载时，YooAsset 认为自己可以卸载资源，但实际上 `ResourceManager` 的引用计数尚未归零，导致**内存峰值居高不下，资源无法被正确卸载**。

解决方案：把 `ResourceManager` 对资源的引用改为**弱引用（WeakReference）** 包装。这样业务模块不需要手动调用 Release —— 当业务模块销毁时，GC 自动回收弱引用，资源计数归零，YooAsset 可以正常卸载资源。

**问题二：图集宽高比失衡导致压缩率极低。**

开发了一个编辑器插件，自动检测宽高比异常的背景图（被严重拉伸），压缩率超过 40% 时报警，然后将这些背景图拆分出独立的图集。

**问题三：废弃资源残留。**

产品侧维护一套有效资源路径表，建立索引后自动剔除已废弃但仍被打包进 AssetBundle 的资源。

---

### Q2：为什么 YooAsset 本身有引用计数，还要自己再维护一套弱引用？

**标准答案：**

YooAsset 的引用计数机制需要**业务层显式通知"我不再使用这个资源了"**，它才会把引用计数减下去。如果业务模块忘记调用 Release，或者业务模块被销毁时没有及时释放，引用计数就不会归零，资源卸载不掉。

自研弱引用层的目的是：**由框架自动感知业务模块的生命周期**，在模块销毁时自动触发 YooAsset 的 Release，不需要每个业务模块都手动管理资源释放，降低了业务层的负担，也减少了因遗漏 Release 导致的内存泄漏风险。

---

### Q3：YooAsset 同步加载方法是怎么实现的？它底层不是异步的吗？

**标准答案：**

YooAsset 对外提供的 `LoadAssetSync<T>` 同步接口，内部**本质上仍然是异步实现**，但通过以下方式模拟出了同步的阻塞效果：

1. **while 循环 + 状态轮询**：在同步方法内部，用一个 `while` 循环持续检测异步操作的完成状态（如 `IsDone` 标志），直到加载完成才让方法返回。
2. **主线程挂起**：在循环等待期间，主线程被阻塞（相当于 busy-wait），因此调用方感受到的是"同步"行为。

**核心原理**：

```csharp
// YooAsset 同步加载简化原理
public T LoadAssetSync<T>(string address) where T : Object
{
    var handle = LoadAssetAsync<T>(address);
    while (!handle.IsDone) // 轮询等待完成
    {
        // 主线程阻塞在这里
    }
    return handle.Result;
}
```

**代价**：主线程被阻塞会导致游戏卡顿，尤其在资源较大或涉及网络下载时。在 WebGL/Android 等平台，同步加载甚至可能直接导致**卡死**（详见 Q21）。因此 YooAsset 官方强烈建议仅在资源已在本地且加载极快时使用同步接口。

### Q4：AssetBundle 中多个 Bundle 共用资源的处理方式是什么？

**标准答案：**

Unity 在打包时会**自动将跨 Bundle 共用的资源抽取到一个独立的共享 Bundle（SharedBundle）** 中，其他所有引用该资源的 Bundle 都会声明对它的依赖。这样：

- 同一份资源只存储一次，节省包体大小
- 运行时内存中只有一份实例，避免重复加载
- 卸载时需要正确处理共享 Bundle 的生命周期，不能在仍有依赖方时提前卸载

---

### Q5：优化打包规格、降低资源冗余具体是怎么做的？

**标准答案：**

三个方向：

1. **废弃资源清理**：产品维护有效资源路径表，建立索引后剔除已废弃但仍被打包的资源。

2. **图集宽高比优化**：编写插件检测宽高比异常的图集（如背景图被严重拉伸），压缩率超过 40% 时报警，然后拆分出独立图集，减少无意义的像素浪费。

3. **MIP 层级优化**：对于不需要代码动态加载的静态图片（如 Logo、启动图），在 Inspector 中禁用 `Generate Mip Map` 选项 —— 这样可以缩短 MIP 链长度，既减少包体，又缩短加载时间。

## 二、UI 框架设计

### Q6：设计一个基于 UGUI 的 UI 框架，核心结构是什么样的？

**标准答案：**

**三层分级结构（参考腾讯 UI 框架方案）：**

| 层级 | 用途 | 特点 |
|------|------|------|
| 全屏界面层 | 商店、背包、技能、任务 | 支持三级堆叠，第四层弹出时丢弃最底层，保证用户能回到之前的界面 |
| 弹窗层 | 确认框、提示框 | 全局唯一，依次顶起，后弹的在上 |
| 小提示层 | Tips、飘字 | 随用随销毁，层级最高 |

**通信机制：**

- **同模块内**：直接调用，数据通过事件流向上传递
- **跨模块**：通过 **EventBus 事件总线** 解耦，避免集中式全局广播

**资源释放策略：**

待释放的资源先进入一个**队列**，分帧逐步卸载，防止大量资源同时卸载引发卡顿。

---

### Q7：同一层内多个界面的遮挡顺序是怎么定义的？

**标准答案：**

同层界面的遮挡顺序通过 `Order in Layer` 字段控制。

弹出新界面时，将该界面的 `Order in Layer` 在当前层的基础上加上一个偏移量（如 +1000）。这样：

- 同一层内 A、B、C 按 `Order` 大小依次遮挡
- 偏移量预留了中间的操作空间，避免后续界面弹出时需要重新排布已有的界面
- 例如全屏层共 4 层（1000~4000），每层 1000 的宽度足够容纳多次界面切换

---

### Q8：全屏界面三级堆叠，第四层弹出时丢弃最底层的逻辑是什么？

**标准答案：**

出于性能考虑，同一时刻只保留最近的三层全屏界面。当第四层弹出时，最底下的那层界面被强制关闭（Destroy），同时记录其状态，以便用户在返回时能恢复到正确的上下文。这样既保证了用户能回到之前的界面，又防止 Canvas 下节点过多导致的合批恶化。

---

## 三、粒子特效与渲染层级

### Q9：UI 界面上挂载粒子特效，有哪些处理方式？如何保证渲染顺序？

**标准答案：**

**三种处理方式：**

**方式一（推荐）：粒子与 UI 分离。**

将粒子从 UI 节点中独立出来，放在一个单独的顶层 Canvas 上，UI 元素在下方。粒子不嵌入 UI 层级，UI 可以正常合批，粒子在最外层统一受 Sorting Layer 管理。

**方式二：世界坐标映射。**

将粒子放在场景中一个独立的顶层位置，用道具的**世界坐标实时映射**粒子的位置，实现粒子跟随道具移动的效果，避免粒子嵌入 UI 打断合批。

**方式三：烘焙到序列帧（K 进道具里）。**

将简单发光类特效预烘焙为序列帧图集，作为 Image 的 Sprite 显示。特效直接 K 到道具里，无粒子实例，无合批冲突。

---

### Q10：滚动列表（带 Mask）中粒子特效的裁切问题怎么解决？

**标准答案：**

这是 UI 粒子领域的经典难题。常规粒子系统不受 UGUI Mask 裁切，有以下几种处理思路：

**方案一：RenderTexture 方案（最通用）。**

将粒子和 3D 场景渲染到一个独立的 `RenderTexture`，然后用 RawImage 挂载该纹理。由于 RawImage 是标准 UI 元素，天然支持 Mask 裁切，粒子特效也就被裁切了。缺点是需要额外的相机和渲染开销。

**方案二：UI Particles 插件（如 Particle System 配合自定义 Shader）。**

使用支持 UGUI 的粒子插件，它们会将粒子转换为 UGUI 兼容的顶点数据，写入 Mask 裁切路径。缺点是引入第三方依赖。

**方案三：分层 Canvas + 滚动同步。**

将粒子层单独放在一个独立 Canvas，同时让该 Canvas 的 RectTransform 与滚动列表同步滚动。这种方案无需插件，但实现复杂度较高。

**最佳实践：** 如果粒子特效较为简单（发光、闪烁等），优先使用烘焙序列帧方案；如果是复杂粒子效果且需要 Mask 裁切，使用 RenderTexture 方案。

---

### Q11：滚动列表中，多个 Item 上的粒子如何保证正确的遮挡关系（如 A 粒子不被 B 图片覆盖）？

**标准答案：**

这个问题本质上是"多个动态粒子在动态 UI 列表中如何统一排序"的问题。

**方案：将粒子统一提升到粒子专属的 Sorting Layer / Order。**

给粒子预设分配一个独立的 Sorting Layer，并设置足够高的 Order in Layer 值，确保所有粒子始终渲染在商品图片之上。具体做法：

- 所有粒子预设使用同一个 Sorting Layer（如 "UI_Particle"）
- 在该 Layer 下统一设置 Order（如 5000）
- 商品图片的 Order 设置为 0~4999

这样无论 Item 如何滚动，所有粒子都在同一层叠顺序上互相遮挡（A 粒子始终遮挡 B 图片），不会出现 Item 顺序变化导致的粒子遮挡混乱。

> 注：面试者在被追问滚动列表场景时坦诚表示"这个没有了解"，说明这是一个常见的进阶知识点，需要专项研究。

## 四、渲染批次与性能

### Q12：UGUI 合批（Batching）的规则是什么？

**标准答案：**

**合批的四个必要条件：**

| 条件 | 说明 |
|------|------|
| **同图集** | 所有 Image 必须使用同一个 Sprite Atlas |
| **同 Sorting Layer 和 Order in Layer** | 不同层级不参与合批 |
| **同 Shader 和 Material** | 材质和 Shader 必须相同 |
| **同 Render Queue** | 默认都是 Transparent 队列 |

**打断合批的情况：**
- 不同图集穿插摆放（即使相同 Sorting Layer）
- Image 与 Text 穿插（如 TMP 动态文字与图片交叉）
- 遮挡关系变化（深度顺序改变）

**不打断合批的属性：**
- `fill amount`（填充模式）
- `scale`、`rotation`
- `rectTransform` 的位置、宽高

这些属性改变只影响顶点数据，不影响材质，因此**不影响合批**。

---

### Q13：TMP 文字如何参与合批？

**标准答案：**

TMP 有两种模式，合批表现截然不同：

| 模式 | 说明 | 合批表现 |
|------|------|---------|
| **动态 TMP**（默认） | 文字网格在运行时动态生成，材质独立 | **无法**与其他 UI 元素合批 |
| **静态烘焙 TMP**（Static Font Asset） | 常用字符预先烘焙为图集，视为普通 Sprite | 可以**参与**合批 |

**实现方法**：在 TextMeshPro Font Asset 设置中，选择 `Create Static Font Asset`，将常用字符集烘焙为图集。此后该 Font Asset 下的文字就可以与其他使用同一图集的元素合批了。

---

### Q14：渲染队列（Render Queue）是什么概念？

**标准答案：**

渲染队列是 Shader 中定义的一个整数标签（`RenderQueue`），用来告诉 GPU 这批几何体应该在**哪个阶段被绘制**。Unity 内置的队列包括：

| 队列名 | 值 | 用途 |
|--------|-----|------|
| Background | 1000 | 最先绘制（如天空盒） |
| Geometry | 2000 | 不透明物体 |
| AlphaTest | 2450 | 半透明测试 |
| Transparent | 3000 | 半透明物体（从后往前绘制） |
| Overlay | 4000 | 最顶层 UI |

**UGUI 合批中的 Render Queue**：UGUI 默认使用 `Transparent` 队列。如果两个 UI 元素的 Shader 中 `RenderQueue` 值不同，**即使其他条件都满足也无法合批**。开发者也可以在 Shader 中手动设置 RenderQueue 来控制合批行为。

---

### Q15：卡片边缘呼吸发光效果是怎么实现的？

**标准答案：**

这是典型的**边缘检测 + 时间驱动**特效，原理如下：

1. **UV 坐标提取边缘**：利用 UV 坐标在 0→1 之间的渐变特性，取 `uv.x`、`uv.y`、`1-uv.x`、`1-uv.y` 的最小值，得到的就是四个方向上的边缘距离。
2. **边缘掩码（01 值）**：用一个 `smoothstep` 将边缘区域提取出来，得到只包含轮廓的掩码图。
3. **呼吸动画**：用一个 `sin(time)` 时间函数驱动发光颜色的强度，叠加到基础颜色上。
4. **菲涅尔效果（可选）**：在 3D 物体上可以用视图方向做菲涅尔加权，让边缘更亮、中心更暗。

```hlsl
// 简化的边缘发光逻辑
float edge = min(min(uv.x, uv.y), min(1.0 - uv.x, 1.0 - uv.y));
float mask = smoothstep(0.0, 0.1, edge);
float breathe = sin(_Time.y * _Speed) * 0.5 + 0.5;
float3 glow = _GlowColor * mask * breathe;
o.color = _BaseColor + glow;
```

### Q16：图形学基础在 UI 开发中的实际应用？

**标准答案：**

- 理解**渲染管线**：顶点着色器 → 几何处理 → 光栅化 → 片段着色器，能帮助你写出高效的 Shader
- **RenderQueue / Sorting Layer / Order in Layer** 构成的三级排序体系，是 UGUI 粒子层级问题的核心
- **Depth Buffer / Stencil Buffer**：在 UI 上做复杂效果（如 UI 内嵌 3D 模型）时，必须理解深度缓冲的工作方式
- **SRP（Scriptable Render Pipeline）**：通过自定义 SRP，可以精细控制 UGUI Canvas 的渲染顺序和后处理

## 五、系统解耦

### Q17：模块之间的解耦原则是什么？什么时候用事件，什么时候直接调用？

**标准答案：**

**判断标准——依赖关系决定通信方式：**

| 场景 | 推荐方式 |
|------|---------|
| A 模块功能强依赖 B 模块（B 消失 A 无意义） | **直接调用**（紧耦合可接受） |
| A 模块与 B 模块相互独立（可独立工作） | **事件总线 / 消息中心** 解耦 |

**解耦原则：**

- 事件只描述"发生了什么"（如 `EventCodes.ItemUsed`），不携带具体业务逻辑
- 同模块内通过事件流向上传递数据，**禁止**跨模块直接引用或全局广播
- 避免形成环形依赖（A→B→C→A）
- 事件订阅/取消订阅要成对，避免内存泄漏

## 六、热更新方案

### Q18：对 HybridCLR 热更新方案了解多少？

**标准答案：**

HybridCLR 是 Unity 目前最成熟的 C# 热更新方案（对标 iOS 的 `roll_ahead` 不可变热更新），核心流程：

```
C# 源码 → IL 中间语言
    ↓ HybridCLR（Huatuo 解释器）
IL → IL2CPP 编译层
    ↓
运行时（原生代码执行）
```

**xLua 方案流程（王者荣耀世界使用）：**

```
C# 逻辑 → Lua 脚本
    ↓ xLua 运行时
Lua → Lua C binding
    ↓ 编译层
IL2CPP
    ↓
运行时
```

两者本质都是"解释执行 + IL2CPP 兜底"：HybridCLR 直接解释 C# IL，不需要额外语言，学习成本低；xLua 需要写 Lua，但生态成熟，在王者荣耀等大型项目中验证过。

---

### Q19：Protobuf（原表）在游戏中的用途？

**标准答案：**

Protobuf（协议缓冲区）是 Google 开源的结构化数据序列化协议，在游戏开发中主要用于：

- **配置表序列化**：将策划配置的 Excel/CSV 表格导出为二进制 `.bytes` 文件，运行时通过 Protobuf 解析。相比 JSON/XML，体积更小、解析更快。
- **网络协议**：客户端与服务器之间的数据包序列化，支持多语言、向后兼容。
- **面向对象特性**：Protobuf 支持继承、组合、重载运算符等，可以像面向对象编程一样描述数据结构。

## 七、WebGL / 微信小游戏适配

### Q20：大型游戏移植到微信小游戏做了哪些优化工作？

**标准答案：**

**问题一：包体过大。**

- **代码分包**：IL2CPP 编译后的 wasm 代码包解压后进入内存约 **10 倍**于包体大小，必须拆分。Unity 支持代码分包（Code Sub-packaging），将 wasm 拆为初始包 + 子包，用户先下载小体积主包快速启动。
- **代码裁剪**：IL2CPP Stripping Level 调高；去掉 `get/set` 属性访问器（可减少约 1/3 代码体积）；移除未使用的协程库等基础依赖（可减少约 30%）。

**问题二：运行内存爆表（iOS 上峰值 1600 MB）。**

- **渐进式图片加载插件**：打包时生成所有图片的低清晰度预览图（模糊缩略图），进入引擎后，图片可见时才触发高清资源下载。既减少初始包体，又加快首屏加载。
- **懒加载**：非关键资源延迟到使用时再加载。

**问题三：后端通信。**

- 后端原始使用 TCP Socket，微信小游戏环境只支持 WebSocket，自研适配层将 TCP 转为 WebSocket 通信。

---

### Q21：WebGL 平台同步加载为什么会卡死？

**标准答案：**

Unity WebGL 的文件系统由 **JavaScript 在内存中模拟**，底层 IO 操作本身就是异步的 —— **WebGL 不支持同步文件 IO**。

当调用同步加载接口时：
1. JS 层无法真正阻塞主线程等待文件读取
2. 主线程被强制挂起等待一个"永远等不到"的同步结果
3. 游戏表现为**卡死（假死）**，用户无感知，以为游戏崩溃了

**根本原因**：这不是 YooAsset 或 Unity 的实现问题，而是 WebGL 平台的底层限制。正确做法是**将所有资源加载改为异步 API**。

> 面试者回答"同步会卡死，因为 WebGL 不支持同步"是对的，但被追问"具体是实现方式的问题还是什么"时未能深入回答。资深候选人应理解：这是平台底层限制，不是 YooAsset 的 bug，而是 API 设计层面的取舍。

## 八、MVVM 框架

### Q22：MVVM 框架在 UI 仿真项目中是怎么用的？

**标准答案：**

MVVM（Model-View-ViewModel）是 UI 系统中广泛使用的架构模式：

- **Model**：数据层，对应业务逻辑和数据结构
- **View**：视图层，对应 UGUI 的 Prefab 和界面展示
- **ViewModel**：视图模型，连接 View 和 Model，负责数据绑定、命令转发、界面状态管理

在 UI 仿真系统中，ViewModel 负责：
- 将 Model 层的数据变化**双向绑定**到 View 的 UI 控件上
- 处理 View 侧的用户操作（点击、输入），转发为命令到 Model
- 管理界面的打开/关闭、显示/隐藏等状态

这套框架使得 UI 逻辑（ViewModel）与 UI 表现（View）完全解耦，策划可以改界面样式，程序改数据逻辑，互不干扰。

## 九、AI 辅助编程

### Q23：AI 编程的工作流是什么样的？

**标准答案：**

**完整工作流：**

1. **分层索引设计**（大前提）：
   - 项目代码量过大（如 1000 万行），AI 无法一次性理解全貌
   - 在项目根目录下维护两个核心文件：项目索引（按模块划分）和术语对照表
   - 让 AI 每次只处理单个模块的上下文

2. **术语文档对齐**：
   - 每个业务模块编写一份小文档，描述功能点、术语含义、对应的代码路径
   - 沟通前先让 AI 读取该文档，确保术语理解一致

3. **需求细化流程**：
   - 用 **Plan 模式**（Agent）先设计方案，AI 会主动提问以细化需求
   - 需求明确后，用 **Agent 模式**生成代码
   - 让 AI **同时生成单元测试用例**，自测通过后才输出

4. **Skills 封装**：
   - 将高频操作封装为 Skills（如 `OpenUI`、`SendReward`、`LoadList`）
   - AI 调用时直接引用，避免每次重复描述

5. **文件行数控制**：
   - 单个文件控制在 **800~1000 行以内**（死线 1000 行）
   - 保证 AI 在上下文窗口内能完整理解代码逻辑

### Q24：AI 生成 1000 万 Token 级别代码接近零返工，是怎么做到的？

**标准答案：**

核心在于**需求清晰度 + 上下文质量**：

1. **通过大量使用总结了 AI 的"易踩坑模式"**，在提需求时有针对性地规避
2. **分层索引做得足够细**：AI 不需要扫描全库，每次只"读"相关模块，大幅降低读取消耗（`token` 消耗分为读取消耗和输出消耗，索引优化主要降低读取消耗）
3. **单次输出约 3000 行代码**，在上述条件下，需求清晰时质量稳定
4. **配合 Skills 封装**：业务模块的高频操作固定模板，AI 生成成功率极高

### Q25：给大需求时，用什么 Agent 模式？

**标准答案：**

| 模式 | 适用场景 |
|------|---------|
| **Plan 模式** | 需求不清晰，需要 AI 先设计方案、提问细化时 |
| **Agent 模式** | 需求已明确，直接生成代码 |
| **Question 模式** | 文档扔进去直接细化，单次生成内完成所有工作 |

**策略**：先 Plan 细化方案 → 再 Agent 生成代码。避免直接丢给 Agent 导致过度设计和凭空捏造。

常用工具：Claude（Codex 内置）、Cursor Agents、GPT-4.6/4.7 等主流模型均有类似模式。

### Q26：如何把控 AI 生成代码的工程化质量？

**标准答案：**

- **架构渐进式**：从小处着手，先做最小可运行版本，遇到瓶颈时再重构拆分架构（一开始拆三层，后来遇到问题才拆四层）。而不是一开始过度设计。
- **核心逻辑人工 review**：AI 擅长业务模块生成，但架构设计和核心算法必须人工把关。AI 擅长通过测试用例编程，容易"为了通过测试而设计"，导致架构本末倒置。
- **多 Agent 协作**：一人扮演产品经理 Agent 出方案，一人扮演架构师 Agent 设计（不写代码），多人负责实现和测试 —— 分工明确但 token 消耗高，适合探索性项目。
- **定期 revive / review**：代码量膨胀后，AI 输出质量下降，必须定期人工清理和重构代码。

### Q27：纯 UI 项目如何提升技术天花板？

**标准答案：**

- **不局限于 UI**：学习战斗模块、寻路系统等核心游戏逻辑，提升全栈能力
- **往底层走**：资源管理底层（YooAsset/Addressables 源码）、渲染批次原理、Shader 编写
- **工具链建设**：开发提升团队效率的工具（插件、自动化流水线）
- **AI 赋能**：将 AI 融入日常工作流，形成工具 + 方法论的系统性竞争力
